# Как раскладывать файлы в репозитории

Используй эту структуру, если в репозитории нужно хранить продуктовую и архитектурную документацию по существующему продукту.

## Базовая структура

```text
repo/
├── README.md                         # Точка входа в репозиторий и правила работы с ним
├── features-index.md                # Плоский реестр текущих фич с кратким описанием и ссылками
├── glossary.md                      # Термины и определения системы
├── open-questions.md                # Открытые вопросы и пробелы, требующие закрытия
├── features/                        # Продуктовые и бизнесовые описания текущих фич
│   └── 0001-example-feature.md      # Одна фича или capability в одном файле
├── architecture/                    # Архитектурные артефакты по текущему состоянию системы
│   ├── constraints.md               # Ограничения системы и реализации
│   ├── requirements.md              # Наблюдаемые или выведенные ФТ и НФТ
│   ├── hld.md                       # High-level design текущего решения
│   ├── landscape.yaml               # Карта сервисов, хранилищ и связей
│   ├── roles-and-permissions.md     # Роли и матрица функционала
│   ├── tech-stack.md                # Ключевые технологии и архитектурно значимые зависимости
│   ├── integrations/                # Описание внешних и межсистемных интеграций
│   │   ├── overview.md              # Обзор всех интеграций
│   │   └── <service>.md             # Один файл на сервис со всеми входящими и исходящими интеграциями
│   ├── contracts/                   # Машиночитаемые контракты интерфейсов
│   │   ├── <sync-api>.yml           # OpenAPI для синхронного API
│   │   └── <async-events>.yml       # AsyncAPI для событий и асинхронных взаимодействий
│   └── storage/                     # Описание БД, кэшей, топиков и других хранилищ
│       └── <storage>.yml            # Один файл на одно хранилище или логическую схему
```

Шаблоны не нужно хранить в продуктовой документации. Их надо брать из самого skill и использовать только для генерации или заполнения целевых файлов.

## Как раскладывать документы

## Как ссылаться на исходный код

Если в документации, feature-описаниях, HLD, glossary, ролях, интеграциях или вопросах на уточнение ты ссылаешься на исходный код, всегда указывай путь вместе с репозиторием-источником.

Правильно:

- `frontend-web/src/app/[locale]/news/[id]/page.tsx`
- `billing-service/internal/app/usecases/create_invoice.go`

Неправильно:

- `src/app/[locale]/news/[id]/page.tsx`
- `internal/app/usecases/create_invoice.go`

Это обязательное требование, потому что архитектурный репозиторий собирается по нескольким репозиториям, и путь без имени репозитория не позволяет надежно перейти к источнику факта.

### `features/`

Сюда клади документы по уже существующим capability продукта.

Не привязывайся к структуре кода. Привязывайся к бизнес-сценариям и пользовательским потокам.

Примеры:

- `0001-customer-onboarding.md`
- `0002-order-processing.md`
- `0003-notification-delivery.md`

### `features-index.md`

Это корневой реестр текущих фич.

Храни здесь:

- название фичи
- короткое описание
- ссылку на основной файл в `features/`

Обновляй его по мере изучения каждого репозитория, а не только в конце анализа.

### `architecture/constraints.md`

Сюда клади выявленные ограничения текущей системы:

- технические ограничения
- организационные ограничения
- инфраструктурные ограничения
- регуляторные ограничения

### `architecture/requirements.md`

Сюда клади системные требования, которые можно наблюдать или обоснованно вывести:

- availability
- performance
- security
- observability
- reliability

Если требования неизвестны, лучше явно написать, что они не подтверждены артефактами, чем выдумывать их.

### `architecture/hld.md`

Сюда клади высокоуровневое описание текущего решения:

- компоненты
- границы ответственности
- основные потоки
- зависимости
- архитектурные риски

### `glossary.md`

Сюда выноси:

- термины, непонятные агенту
- термины, подсвеченные пользователем
- термины, которые в системе трактуются иначе, чем агент ожидал
- доменные понятия, которые стоит зафиксировать для дальнейшего анализа

Для каждого термина фиксируй определение и источник подтверждения.

### `open-questions.md`

Это корневой файл со всеми открытыми вопросами по продукту и архитектуре.

Храни здесь:

- текст вопроса
- связанный репозиторий, если он есть
- текущий статус
- нужен ли ответ пользователя
- что уже известно
- чем вопрос был закрыт или что еще нужно для закрытия

При анализе каждого нового репозитория агент должен пересматривать уже открытые вопросы и закрывать те, для которых нашлось подтверждение в коде, тестах, конфигурации или других артефактах.

### `architecture/integrations/`

Сюда клади описания всех заметных интеграций.

Используй:

- `overview.md` как обзорную карту
- отдельные файлы по одному на сервис, где перечислены все входящие и исходящие интеграции этого сервиса

### `architecture/roles-and-permissions.md`

Сюда клади:

- список ролей, найденных в системе
- краткое описание роли
- матрицу функционала: какая роль что может делать
- ограничения и условия доступа

Обновляй этот файл по мере появления новой информации из каждого репозитория, особенно если находятся guards, policies, permission checks, role enums, ACL или UI-ограничения.

### `architecture/contracts/`

Сюда клади контракты, если их можно восстановить или извлечь:

- OpenAPI для синхронных интерфейсов
- AsyncAPI для событий

Если контракты нельзя восстановить полностью, допускается создать черновик и явно отметить неполные места.

### `architecture/storage/`

Сюда клади структуру данных:

- таблицы
- коллекции
- индексы
- топики
- ключи кэша
- поля и назначения

### `architecture/landscape.yaml`

Сюда клади карту сервисов, внешних систем, хранилищ и связей между ними.

Этот файл должен отвечать на вопрос, из чего состоит продукт как техническая система.

Для каждого сервиса фиксируй краткое поле `technology` в формате `<runtime> / <framework>`.

### `architecture/tech-stack.md`

Сюда клади детальную сводку по ключевому стеку сервисов.

Используй этот файл для архитектурно значимых технологий:

- язык и runtime
- основной framework
- ORM и доступ к данным
- transport и integration libraries
- broker clients
- observability
- security и auth библиотеки
- важные внешние SDK

Не превращай этот файл в выгрузку всех пакетов из lockfile. Фиксируй только то, что влияет на понимание архитектуры и сопровождение системы.

## Порядок наполнения

1. Сначала собери список репозиториев в scope и зафиксируй предполагаемую роль каждого.
2. Затем проходи репозитории по одному и после каждого прохода обновляй `architecture/hld.md`, `architecture/landscape.yaml`, `architecture/tech-stack.md`, `glossary.md`, `open-questions.md`, `architecture/roles-and-permissions.md` и `features-index.md`.
3. По мере появления фактов выноси интеграции в `architecture/integrations/`.
4. По мере появления подтверждений собирай контракты в `architecture/contracts/`.
5. По мере появления структуры данных описывай хранилища в `architecture/storage/`.
6. По мере появления новых capability создавай или уточняй документы в `features/`.
7. После прохода по доступным репозиториям провалидируй согласованность артефактов между собой и с исходными репозиториями.
8. Затем собери список открытых вопросов и пробелов.
9. Затем уточни пробелы у пользователя.
10. Перед завершением еще раз провалидируй итоговую картину.
