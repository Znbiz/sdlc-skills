# Как восстанавливать текущее состояние продукта

Этот workflow нужен, когда система уже работает, и нужно разложить ее по feature- и architecture-артефактам.

Перед началом создай и веди файл прогресса анализа по шаблону `assets/repo-initialization-progress-template.yaml`. Обновляй его после каждого проанализированного репозитория, после каждой валидации и после каждого интервью с пользователем.

## Шаг 1. Определи границы анализа и список репозиториев

Зафиксируй:

- какой продукт и какие репозитории анализируются
- это один сервис или платформа из нескольких сервисов
- есть ли явные внешние зависимости
- что считается in scope

До начала анализа запроси у пользователя полный список репозиториев в scope. Если списка нет, не начинай анализировать соседние папки, текущий workspace или случайные локальные checkout'ы по собственной инициативе.

После того как пользователь предоставил список репозиториев:

- создай локальный каталог `.temp/` в текущем workspace, если его еще нет
- добавь `.temp/` в `.gitignore`, если его там еще нет
- клонируй все репозитории из пользовательского списка именно в `.temp/`
- зафиксируй, какие именно checkout'ы в `.temp/` стали источниками фактов

Если пользователь отдельно попросил использовать уже существующую локальную копию репозитория, зафиксируй это как явное исключение. Без такого подтверждения не считай соседнюю папку источником `as-is`.

Для каждого склонированного или явно подтвержденного локального репозитория перед анализом:

- переключись в их основную ветку, например `main` или `master`
- обнови ее до актуального коммита из удаленного репозитория
- зафиксируй, какая именно ветка и какой commit SHA стали базой анализа

Не анализируй случайную локальную feature-ветку как `as-is` состояние продукта, если это отдельно не оговорено.

## Шаг 2. Спланируй проход по репозиториям

Не пытайся сразу описать всю систему одним большим заходом. Для multi-repo продукта рабочий порядок такой:

- собери полный список репозиториев
- зафиксируй предполагаемую роль каждого репозитория
- выбери порядок обхода: обычно от entrypoint-репозиториев к библиотекам и инфраструктурным зависимостям
- после анализа каждого репозитория сразу обновляй общую архитектурную картину

Идея в том, чтобы картина системы нарастала репозиторий за репозиторием, а не собиралась в конце по памяти.

## Шаг 3. Построй первичный технический срез по репозиториям

Сначала определи:

- какие сервисы, библиотеки и джобы входят в систему
- как они связаны между собой
- где проходят синхронные и асинхронные границы
- какие репозитории содержат контракты, схемы, миграции и integration points

Этот шаг нужен, чтобы сначала наполнить `architecture/`, а не сразу уходить в описание фич.

## Шаг 4. Обновляй общую архитектурную картину после каждого репозитория

После анализа каждого репозитория дополняй и уточняй:

- `architecture/hld.md`
- `architecture/landscape.yaml`
- `architecture/tech-stack.md`
- `architecture/integrations/`
- `architecture/contracts/`
- `architecture/storage/`

Не жди конца анализа всех репозиториев. Если новый репозиторий меняет понимание картины, сразу отражай это в артефактах.

## Шаг 5. Найди основные входы в систему

Обычно это:

- HTTP API
- gRPC API
- события из брокера
- cron/scheduler jobs
- batch-процессы
- административные интерфейсы

Эти точки входа помогают понять, какие capability реально существуют.

## Шаг 6. Собери архитектурные артефакты

До интервью с пользователем обновляй:

- `architecture/hld.md` для общей архитектурной картины
- `architecture/integrations/` для внешних и межсервисных взаимодействий
- `architecture/contracts/` для API и событий
- `architecture/storage/` для БД, кэшей, топиков, файловых хранилищ
- `architecture/landscape.yaml` для карты компонентов и связей между ними
- `architecture/tech-stack.md` для таблицы по ключевому стеку и архитектурно значимым зависимостям

Для `architecture/integrations/` не ограничивайся общими словами. Постарайся вытащить все внешние интеграции, которые читаются из кода и конфигурации:

- внешние HTTP/gRPC API
- брокеры и внешние топики
- S3/SFTP и файловые шлюзы
- сторонние SDK и platform services
- webhook endpoints и callback-интеграции

Каждую значимую внешнюю интеграцию фиксируй в отдельном файле, а общую картину своди в `overview.md` по шаблону `assets/architecture/integrations-overview-template.md`.

Для `architecture/landscape.yaml` фиксируй у каждого сервиса поле `technology` в формате `<runtime> / <framework>`, например `python3.12 / fastapi` или `node20 / nestjs`.

Для `architecture/tech-stack.md` собирай не полный список всех пакетов, а только ключевые зависимости и технологии:

- язык и runtime
- основной framework
- ORM или data access layer
- transport и API libraries
- broker или queue clients
- observability stack
- security и auth библиотеки, если они значимы
- архитектурно важные внешние SDK

Для каждой записи фиксируй сервис, зависимость, версию, роль и источник подтверждения.

## Шаг 7. Выдели доменные сценарии

Не группируй по техническим слоям. Группируй по смыслу:

- регистрация клиента
- оформление заказа
- расчет тарифа
- выдача уведомления
- смена статуса сущности

Каждый такой сценарий может стать отдельным документом в `features/`.

## Шаг 8. Построй поток по каждому сценарию

Для каждого ключевого сценария ответь:

- кто инициирует
- какой вход приходит
- какие проверки выполняются
- какие сервисы вызываются
- какие данные читаются и пишутся
- какие события публикуются
- какие ошибки и fallback-сценарии есть

## Шаг 9. Проведи валидацию промежуточной картины

После прохода по доступным репозиториям проверь:

- согласованы ли `architecture/hld.md` и `architecture/landscape.yaml`
- согласован ли `architecture/tech-stack.md` с `landscape.yaml`, lockfile и manifest-файлами
- все ли описанные интеграции отражены в общей картине и детализации
- не противоречат ли контракты хендлерам, DTO, схемам и тестам
- не расходятся ли хранилища и структуры данных с миграциями и моделями
- нет ли утверждений без опоры на конкретные артефакты

Если валидация вскрыла расхождения, сначала исправь артефакты и только потом переходи дальше.

## Шаг 10. Сначала собери открытые вопросы

После прохода по доступным репозиториям отдельно выпиши:

- что не найдено в коде
- что противоречит другим артефактам
- что выглядит неоднозначным
- что требует бизнес-подтверждения, а не технической реконструкции

Только после этого переходи к вопросам пользователю.

## Шаг 11. Проведи интервью по пробелам

После первичного анализа кода и заполнения `architecture/` переходи к вопросам пользователю. Спрашивай только то, что нельзя надежно восстановить из артефактов:

- где проходит реальная бизнес-граница фичи
- какие роли и пользователи участвуют
- какие ручные операции, исключения или регламентные шаги не отражены в коде
- какие сценарии считаются критичными для бизнеса, хотя в коде они размазаны

Если пользователь утверждает, что какое-то поведение, правило, интеграция или сценарий уже есть в системе:

- не считай это автоматически подтвержденным фактом
- сначала проверь, находится ли этому подтверждение в коде, конфигурации, тестах, контрактах или runtime-артефактах
- если подтверждение не найдено, явно скажи пользователю, что пока не видишь этого в реализации
- после этого попроси указать, где именно это реализовано: в каком репозитории, сервисе, модуле, тесте, конфиге или сценарии
- если пользователь не может указать реализацию и ты тоже не находишь подтверждения, не включай это в `as-is` описание и помечай как неподтвержденное

Если пользователь описывает желаемое или ожидаемое поведение, а не подтвержденную реализацию, не смешивай это с текущим состоянием.

## Шаг 12. Дополни `features/` и требования

После интервью обновляй:

- `features/` для capability и пользовательских сценариев
- `architecture/requirements.md`, если требования действительно подтверждаются кодом, конфигами или стабильной бизнес-информацией
- связанные архитектурные артефакты, если пользователь помог уточнить назначение уже найденных компонентов

Если пользователь начинает описывать желаемое будущее поведение, не смешивай его с `as-is`: выноси это как отдельный вопрос или передавай дальше в `new-feature-arch-skill`.

## Шаг 13. Проведи финальную валидацию

Перед завершением анализа еще раз проверь:

- нет ли конфликта между `features/` и `architecture/`
- все ли выводы можно отнести к фактам, выводам или предположениям
- не появились ли после интервью утверждения без технического подтверждения
- отражены ли в пробелах все неподтвержденные или спорные места

## Шаг 14. Зафиксируй пробелы

Для brownfield-систем это обязательно.

Отдельно перечисляй:

- что подтверждено кодом
- что выведено косвенно
- что требует интервью с командой
- что выглядит неактуальным или противоречивым

## Практическое правило

Если не удается сразу понять продукт по коду, сначала запроси у пользователя полный список репозиториев и собери их в `.temp/`, затем восстанови карту репозиториев, пройди ключевые репозитории, собери 2-3 главных end-to-end сценария и проверь, что они согласуются с артефактами. Обычно этого достаточно, чтобы потом корректно разложить остальную систему.
