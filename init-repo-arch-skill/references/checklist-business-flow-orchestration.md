# Пункт checklist: business_flow_orchestration

Собрать слой оркестрации: use cases, application services, state transitions — то, что описывает бизнес-логику, а не технические детали инфраструктуры.

## Что искать

- **Application services / use case classes** — классы или модули с именами вида `*Service`, `*UseCase`, `*Interactor`, `*Handler`, `*Command`
- **Orchestration code** — код, который последовательно вызывает несколько других сервисов или шагов для выполнения одного сценария
- **State transitions** — конечные автоматы, статусы сущностей (`OrderStatus`, `PaymentState`), переходы между ними
- **Domain events** — внутренние события, которые генерирует система при изменении состояния
- **Transaction boundaries** — где открываются транзакции, что в них входит
- **Error handling flows** — как система реагирует на ошибки: retry, compensation, rollback

## Шаги выполнения

1. Найти `services/`, `use_cases/`, `application/`, `domain/`, `usecases/` директории.
2. Для каждого крупного сценария (регистрация, оплата, отправка и т.п.) восстановить последовательность шагов: кто вызывает кого, в каком порядке.
3. Найти state machine / enum со статусами и зафиксировать список состояний и возможные переходы.
4. Отметить, где логика оркестрации перемешана с инфраструктурным кодом (это архитектурный риск).

## Обязательные выходы

- В `notes`: основные сценарии и краткая схема их оркестрации.
- Если найдена нетривиальная state machine — зафиксировать состояния и переходы.

## Проверка согласованности и правка артефактов

Перед тем как пометить пункт `completed`:

- Обновить раздел **"Основные потоки"** в `architecture/hld.md` — добавить потоки, которые стали известны из этого репозитория.
- Если найдены новые доменные сущности — проверить, упомянуты ли они в `architecture/glossary.md`. Недостающие внести в пункт `glossary_updates`.
- Если state machine сложная и нигде не задокументирована — открыть вопрос в `open-questions.md` с пометкой `ask_user=false`.
- Если оркестрация разделена между несколькими сервисами — пометить в `notes` для последующего обновления `architecture/hld.md` после анализа всех сервисов.
