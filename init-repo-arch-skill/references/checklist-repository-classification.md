# Пункт checklist: repository_classification

Определить тип репозитория, его роль и категорию (`product` или `support`) до начала детального анализа. Это решение определяет, по какому пути дальше идёт весь чеклист этого репозитория.

## Шаги выполнения

1. Прочитать `README.md`, корневой `package.json` / `pyproject.toml` / `go.mod` / `pom.xml` / `build.gradle` — выяснить название, назначение, краткое описание.
2. Определить **тип репозитория**:
   - `service` — сервис с собственным процессом (API, worker, backend)
   - `job` — batch-задача или scheduled job без постоянного процесса, реализующая бизнес-логику
   - `monorepo` — несколько сервисов или пакетов в одном репозитории
   - `library` — переиспользуемый пакет без самостоятельного запуска
   - `test-repo` — репозиторий с integration/e2e/acceptance-тестами, без собственной бизнес-логики
   - `infra` — Helm-чарты, Terraform, Ansible, CI/CD конфиги
3. Определить **категорию** по типу:
   - `product` — типы `service`, `job`, `monorepo`. Репозиторий реализует продуктовую логику и должен пройти **весь чеклист как раньше**, без изменений.
   - `support` — типы `library`, `test-repo`, `infra`. Репозиторий не реализует продуктовую логику сам по себе, а обслуживает продуктовые репозитории. Для него действует **сокращённый путь** (см. ниже) и **отдельное место в документации** — `architecture/support-repositories.md`, а не `landscape.yaml` / `features/`.
4. Определить **роль в системе**: что этот репозиторий делает для продукта, как он соотносится с другими уже известными репозиториями.
   - Для `test-repo` — зафиксировать, какие сервисы и потоки он покрывает (по конфигу тестов, target URL, fixtures, README).
5. Зафиксировать тип, категорию и роль в `notes` пункта `repository_classification`.

## Обязательные выходы

- Поле `notes` заполнено: тип репозитория, категория (`product`/`support`) и роль. Для `test-repo` — дополнительно список покрываемых сервисов/потоков.

## Если категория `product` (service / job / monorepo)

Ничего не меняется относительно прежнего поведения:

- Добавить предварительную запись в `architecture/landscape.yaml`: поле `name`, `role`, `technology` (заполнить частично, дополнить позднее). Пометить в notes, что запись предварительная.
- Если тип `monorepo` — зафиксировать список компонентов и запланировать отдельный проход по каждому как по самостоятельному сервису.
- Если роль репозитория противоречит ранее собранной картине (например, в HLD он не упомянут) — открыть вопрос в `open-questions.md`.
- Продолжить детальный анализ по всем оставшимся пунктам чеклиста в обычном порядке.

## Если категория `support` (library / test-repo / infra)

Не заводить запись в `landscape.yaml` и не создавать `features/` для этого репозитория — это привело бы к смешиванию продуктовой и инфраструктурной/тестовой документации. Вместо этого:

1. Создать или обновить запись в `architecture/support-repositories.md` по шаблону `assets/architecture/support-repositories-template.md`: тип, роль, что обслуживает (список сервисов/доменов), ключевые технологии.
2. Пройти **только применимые** пункты оставшегося чеклиста — см. таблицу ниже. Для пунктов, отмеченных «не применимо», не проводить анализ: сразу зафиксировать пункт как `completed` с notes `не применимо: <тип репозитория>` через `repo --checklist-item <item> --checklist-status completed --notes "не применимо: <тип>"`.
3. В пунктах `architecture_artifact_updates` и `feature_discovery_and_updates` / `features_index_updates` ссылаться на `support-repositories.md`, а не на `landscape.yaml` / `features-index.md` (если репозиторий не несёт продуктовой capability — см. примечание ниже).

## Проверка согласованности и правка артефактов

Перед тем как пометить `repository_classification` `completed`:

- Категория (`product`/`support`) явно зафиксирована в notes — без неё нельзя определить дальнейший путь по чеклисту.
- Если категория `support` — запись добавлена или запланирована в `architecture/support-repositories.md`, а не в `landscape.yaml`.
- Если в процессе дальнейшего анализа репозитория обнаруживается, что классификация была ошибочной (например, в `library` найдены HTTP-handlers с собственной бизнес-логикой) — пересмотреть тип и категорию, обновить notes этого пункта.

## Подводные камни

- Не считай README источником истины, если код и конфигурация говорят обратное.
- Не путай текущее поведение и желаемое будущее состояние: фиксируй только то, что реализовано сейчас.
- Не помечай пункт «не применимо» без короткого обоснования — пустой notes не считается выполненным шагом.
- Технические слои (`api/`, `domain/`, `infrastructure/`) внутри `library` не делают её `service`: категория определяется наличием самостоятельного продуктового процесса, а не структурой кода.
