# Источники фактов для анализа существующего продукта

Используй этот список, чтобы восстанавливать реальную архитектуру системы по сильным сигналам, а не по догадкам.

## Сильные источники

Этим источникам доверяй в первую очередь:

- маршруты API, gRPC handlers, message consumers and producers
- интеграционные тесты и e2e-тесты
- миграции БД и схемы таблиц
- машиночитаемые контракты: OpenAPI, AsyncAPI, protobuf
- deployment-конфигурации, Helm values, Terraform, compose, k8s manifests
- код оркестрации бизнес-процесса

Если сервис можно безопасно поднять локально, используй также runtime-наблюдение как сильный дополнительный источник фактов:

- запуск сервиса или набора сервисов в Docker или Docker Compose
- доступные health/readiness endpoints
- реально поднятые HTTP/gRPC endpoints
- фактические топики, очереди, фоновые процессы и зависимости, которые проявляются при старте
- runtime-конфигурация, если она читается из локального окружения и контейнерных настроек

Если интеграционные или e2e-тесты вынесены в отдельный репозиторий, запрашивай и анализируй этот репозиторий так же, как основные сервисные репозитории. Для восстановления текущего поведения он часто так же важен, как и код самого сервиса.

## Средние источники

Эти источники полезны, но требуют перепроверки:

- README и markdown-документы
- `.env.example`, sample configs
- unit-тесты
- диаграммы, если нет признаков актуальности

## Слабые источники

Используй только как вспомогательный сигнал:

- названия папок без кода
- комментарии без подтверждения в логике
- устаревшие wiki-описания

## Runtime-наблюдение

Когда это возможно и безопасно:

- поднимай сервисы локально в Docker или Docker Compose
- используй запуск как способ подтвердить текущее поведение, состав зависимостей и реальные точки входа
- сверяй runtime-наблюдения с кодом, контрактами и конфигурацией, а не подменяй ими анализ кода

Runtime-наблюдение особенно полезно, когда нужно:

- подтвердить, какие сервисы действительно стартуют вместе
- увидеть фактические порты, endpoints и health checks
- проверить, какие зависимости обязательны для запуска
- понять, какие конфиги и переменные окружения реально используются
- обнаружить скрытые фоновые джобы, consumers, producers или инфраструктурные зависимости

Не считай успешный старт контейнера автоматическим доказательством корректности бизнес-логики. Это источник подтверждения runtime-картины, а не замена анализа кода.

### Фичи

Ищи:

- точки входа пользователя или внешнего клиента
- бизнес-операции и статусы
- use case и orchestration-код
- тесты на end-to-end сценарии
- отдельный репозиторий с e2e или интеграционными тестами, если такие тесты вынесены из основного сервиса

### Интеграции

Ищи:

- HTTP clients
- Kafka/Rabbit/NATS producers and consumers
- S3/SFTP connectors
- внешние SDK
- retry/circuit breaker/timeouts

### Контракты

Ищи:

- OpenAPI/Swagger
- protobuf
- AsyncAPI
- DTO/serializers/request-response models
- message payload builders and parsers

### Хранилища

Ищи:

- ORM models
- SQL migrations
- repository implementations
- cache keys and TTL logic
- topic names and event retention settings

## Правило фиксации

Если утверждение не подтверждено явным артефактом, помечай его как:

- `выведено косвенно`
- `требует подтверждения`
- `не подтверждено кодом`
