# Пункт checklist: data_and_storage

Восстановить модель данных и все хранилища сервиса: БД, кэши, файловые хранилища.

## Применимость

**Если репозиторий — чистый фронтенд** (нет серверного кода, только UI: React/Vue/Angular/Next без API routes, мобильное приложение, статика) — пропустить этот пункт, написать `"frontend — хранилища не применимы"` в notes и перейти к следующему пункту.

Признаки фронтенда: нет migrations/, нет ORM-зависимостей, нет серверных env-переменных для БД, нет docker-compose с базами данных.

**Если репозиторий — бэкенд или фуллстек** — выполнить все шаги ниже с максимальной детализацией.

## Что искать

- **ORM-модели** — `models.py`, `*.entity.ts`, `*Entity.java`, `*Model.go`
- **SQL migrations** — `migrations/`, `db/migrate/`, `flyway/`, `liquibase/`, `alembic/`
- **Repository классы** — паттерн Repository, DAO
- **Cache** — Redis keys, TTL, `@Cacheable`, `cache.set(key, ...)`, cache eviction
- **Topics** — Kafka топики как хранилище (retention, compaction) — но **не** контракты (те в contracts_and_schemas)
- **Event retention** — как долго хранятся события, есть ли event sourcing
- **File storage** — S3 buckets, GCS buckets, локальные пути с файлами, MinIO

## Шаги выполнения

1. Прочитать все ORM-модели и SQL migrations — восстановить **финальную схему каждой таблицы**.
2. Для каждой таблицы/коллекции описать **каждое поле**: тип, nullable, default, индексы, FK, и бизнес-смысл поля (что хранится, откуда берётся, зачем нужно). Не пропускать ни одного поля — даже технические (`created_at`, `updated_at`, `deleted_at`, `version`).
3. Зафиксировать связи между таблицами: один-к-одному, один-ко-многим, многие-ко-многим (через join-таблицы).
4. Найти Redis-ключи и TTL-значения (обычно константы или конфигурация). Описать, что хранится под каждым ключом/паттерном ключа.
5. Найти S3/GCS bucket names и паттерны путей к файлам — что именно хранится, кто пишет, кто читает.
6. Создать артефакт с полными схемами.

> **Качество**: артефакт считается готовым, когда по нему можно восстановить DDL без обращения к коду. Каждое поле должно иметь комментарий с бизнес-смыслом. Пустые или скопированные без смысла поля — не принимаются.

## Обязательные выходы

- **Если есть БД, кэш, или файловое хранилище**: создать `architecture/storage/<service>.yml` по шаблону `assets/architecture/storage-template.yml` с полными схемами и комментариями по каждому полю. Путь к файлу зафиксировать в notes.
- Чистый фронтенд → написать `"frontend — хранилища не применимы"` в notes.
- Нет хранилища (бэкенд без персистентности) → написать `"нет хранилища"` в notes.

## Проверка согласованности и правка артефактов

Перед тем как пометить пункт `completed`:

- Проверить, что в `architecture/landscape.yaml` у сервиса упомянуты его хранилища.
- Если в storage обнаружены персональные данные (PII: email, phone, address, payment info) — отметить для пункта `security_and_auth_updates` как sensitivity area.
- Если обнаружены таблицы без индексов на часто запрашиваемых полях или очень большие таблицы без партиционирования — отметить для пункта `risks_and_tech_debt_updates`.
- Если модель данных используется несколькими сервисами (shared database anti-pattern) — зафиксировать это как риск.
- Сверить с `architecture/hld.md`: все хранилища должны быть упомянуты в описании компонента.

## Подводные камни

- Результат не готов, если хранилища упомянуты, но не выделен источник истины: для каждого хранилища должно быть явно указано, какая сущность или сервис им владеет.
