# Пункт checklist: repository_characterization

Определить тип репозитория и его роль в системе до начала детального анализа.

## Шаги выполнения

1. Прочитать `README.md`, корневой `package.json` / `pyproject.toml` / `go.mod` / `pom.xml` / `build.gradle` — выяснить название, назначение, краткое описание.
2. Определить **тип репозитория**:
   - `service` — сервис с собственным процессом (API, worker, backend)
   - `library` — переиспользуемый пакет без самостоятельного запуска
   - `job` — batch-задача или scheduled job без постоянного процесса
   - `test-repo` — репозиторий с integration/e2e/acceptance-тестами
   - `infra` — Helm-чарты, Terraform, Ansible, CI/CD конфиги
   - `monorepo` — несколько сервисов или пакетов в одном репозитории
3. Определить **роль в системе**: что этот репозиторий делает для продукта, как он соотносится с другими уже известными репозиториями.
4. Зафиксировать тип и роль в `notes` progress-файла.

## Обязательные выходы

- Поле `notes` для `repository_characterization` заполнено: тип репозитория и его роль.

## Проверка согласованности и правка артефактов

Перед тем как пометить пункт `completed`:

- Если тип `service` или `job` — добавить предварительную запись в `architecture/landscape.yaml`: поле `name`, `role`, `technology` (заполнить частично, дополнить позднее). Пометить в notes, что запись предварительная.
- Если тип `test-repo` — пометить в progress-файле как источник поведенческих свидетельств; при анализе других репозиториев ссылаться на него в пункте `tests_and_behavior_evidence`.
- Если тип `monorepo` — зафиксировать список компонентов и запланировать отдельный проход по каждому как по самостоятельному сервису.
- Если роль репозитория противоречит ранее собранной картине (например, в HLD он не упомянут) — открыть вопрос в `open-questions.md`.

## Подводные камни

- Не считай README источником истины, если код и конфигурация говорят обратное.
- Не путай текущее поведение и желаемое будущее состояние: фиксируй только то, что реализовано сейчас.
