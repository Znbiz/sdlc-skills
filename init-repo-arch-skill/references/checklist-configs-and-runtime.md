# Пункт checklist: configs_and_runtime

Собрать конфигурационный слой: переменные окружения, runtime-зависимости, инфраструктурные параметры и feature flags.

## Что искать

- **`.env.example` / `.env.template`** — список env-переменных с описанием
- **Конфигурационные файлы** — `config.yaml`, `settings.py`, `application.yml`, `appsettings.json`
- **Docker / Docker Compose** — образы, зависимые сервисы, healthchecks, порты, volumes
- **Helm charts** — `values.yaml`, `Chart.yaml`, templates с env-переменными
- **Runtime prerequisites** — зависимости от баз данных, брокеров, кэшей при старте
- **Feature flags** — `FEATURE_*`, `FF_*`, LaunchDarkly, GrowthBook, Unleash интеграции
- **Startup sequence** — порядок инициализации компонентов, `depends_on`, wait-for-it

## Шаги выполнения

1. Прочитать `.env.example` и все конфигурационные файлы целиком. `.env.example` — средний источник: показывает ожидаемые переменные, но не гарантирует, что они актуальны; сверяй с кодом.
2. Составить список всех env-переменных: имя, назначение, тип хранилища (`.env` / `helm` / `vault` / `db`), обязательность.
3. Определить список runtime-зависимостей: что должно быть запущено до старта сервиса.
4. Найти feature flags и зафиксировать их имена и влияние на поведение.

## Обязательные выходы

- В `notes`: полный список env-переменных с типом хранилища в формате `service-name: .env / helm / vault / db`.
- В `notes`: список runtime-зависимостей (БД, брокер, кэш и т.п.).

## Runtime-наблюдение

Если сервис можно безопасно поднять через Docker или Docker Compose:

- Запусти и проверь фактические runtime-зависимости (что действительно нужно при старте)
- Сверь реальные env-переменные с `.env.example` — расхождения указывают на устаревшую документацию
- Зафиксируй скрытые зависимости, которые не видны из конфигурационных файлов

## Проверка согласованности и правка артефактов

Перед тем как пометить пункт `completed`:

- Если найдены новые runtime-зависимости (БД, брокер, кэш) — проверить, упомянуты ли они в `architecture/hld.md` и `architecture/landscape.yaml`. Добавить недостающие.
- Если найдены feature flags — зафиксировать их в `features/` соответствующих фич в секции "Конфигурация и управление".
- Обновить `architecture/tech-stack.md` по значимым зависимостям из `Dockerfile` / `docker-compose.yml` / `Helm`.
- Если переменных с секретами много (DB_PASSWORD, API_KEY, PRIVATE_KEY) — отметить для пункта `security_and_auth_updates`.
