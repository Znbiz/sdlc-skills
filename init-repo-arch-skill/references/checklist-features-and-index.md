# Пункты checklist: feature_discovery_and_updates + features_index_updates

Выделить capability продукта из текущего репозитория и обновить реестр фич.

Не пытайся описать все экраны или все методы. Выделяй то, что важно архитектурно:

- ключевые пользовательские сценарии
- критичные бизнес-потоки
- основные доменные сущности
- интеграционно значимые процессы

Одна фича = одна осмысленная capability, не случайный кусок кода.

Если репозиторий категории `support` (`library`/`test-repo`/`infra`, см. `repository_classification`) — оба пункта обычно не применимы: такие репозитории не несут продуктовую capability сами по себе. Зафиксируй это в notes и опиши репозиторий в `architecture/support-repositories.md`. Исключение — если support-репозиторий явно реализует продуктовую capability (редкий случай); тогда веди его и здесь, и в `support-repositories.md`.

Если поведение нельзя восстановить из кода — зафиксируй пробел в `open-questions.md` и переходи к интервью с пользователем только после того, как собран технический каркас.

## Часть 1: feature_discovery_and_updates

### Типы capability для выделения

- **Пользовательские** — сценарии, которые инициирует человек через UI или CLI
- **Интеграционные (M2M)** — публичный API (REST/gRPC/GraphQL), webhooks и callbacks, публикуемые события (Kafka-топики, очереди), file exchange (SFTP/S3), batch-интерфейсы, SSO как провайдер — всё, через что с продуктом взаимодействуют внешние системы
- **Операционные** — cron-джобы, автоматические расчёты, ротация данных, retry и reconciliation-процессы

### Шаги выполнения

1. По итогам пунктов 1–8 checklist выписать capability — не по слоям кода, а по бизнес-сценариям.
2. Для каждой новой capability создать файл `features/<feature-name>.md` по шаблону `assets/feature-template.md`.
3. Для существующих `features/` обновить секцию **трассировки реализации**:
   - точки входа: пути в формате `repo/path`
   - ключевые файлы реализации по роли
   - подтверждающие тесты
   - статус реализации: `implemented` / `partial` / `unknown`
4. Обновить секцию **конфигурации и управления**:
   - env vars и конфиги: формат `service-name: .env / helm / vault / db`
   - feature flags с именами и влиянием
5. Проверить качество продуктового описания в feature-файле:
   - не оставлять разделы `Backend entities`, `Frontend entities`, `Точки входа`, `Entrypoints`, `Ключевые сущности`, `Основной поток` в виде списка имён без пояснения
   - для каждой сущности сначала объяснять, что это за объект в домене или реализации, затем зачем он нужен в этой фиче, и только потом перечислять ключевые поля, связи, статусы или таблицы
   - для групп endpoint'ов и маршрутов указывать не только URL, но и назначение группы: CRUD, public read-side, admin-only operations, RPC/side-effect операции, sync с внешней системой
   - если перечисляется несколько связанных сущностей или endpoint'ов, группировать их по ответственности и явно показывать связь с пользовательским сценарием, а не просто с кодовым модулем
   - списки `Подтверждающие артефакты` и `Ключевые файлы реализации` можно оставлять компактными, но продуктовые разделы с сущностями, потоками и точками входа должны быть объяснены человеческим языком

### Правило статуса

- `implemented` — только если есть точка входа **и** хотя бы один тест
- `partial` — есть точка входа, но тест отсутствует или неполный
- `unknown` — нет достаточного подтверждения

## Часть 2: features_index_updates

1. Открыть `features-index.md`.
2. Для каждой новой фичи добавить строку: название, краткое описание (1 предложение), ссылка на файл.
3. Для изменённых фич обновить описание если оно стало неточным.
4. Убедиться, что `features-index.md` начинается сразу с полезного реестра, без служебного вводного абзаца про назначение файла или списка уже проанализированных репозиториев.

---

## Проверка согласованности и правка артефактов

Перед тем как пометить оба пункта `completed`:

- Все фичи в `features-index.md` должны иметь соответствующий файл в `features/`.
- Нет фич со статусом `implemented` без точки входа или теста.
- Если фича связана с конкретным endpoint из `architecture/contracts/` — убедиться, что трассировка ссылается на этот контракт.
- Если фича использует хранилище из `architecture/storage/` — убедиться, что это отражено в секции трассировки.
- Если появились новые операционные фичи (cron, reconciliation) — проверить, упомянуты ли они в `architecture/hld.md`.
- В feature-файлах нет продуктовых разделов, которые сведены к голому списку имён без пояснения смысла и роли в сценарии.

## Подводные камни

- Не создавай фичи по слоям кода; создавай их по бизнес-сценариям и capability.
- Не создавай фичи по именам контроллеров или таблиц, если за ними не стоит бизнес-сценарий.
- Результат не готов, если нет связи между фичами и техническими компонентами: у каждой фичи должна быть трассировка реализации (точки входа, ключевые файлы, тесты).
