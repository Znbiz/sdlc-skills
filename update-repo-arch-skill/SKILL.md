# Навык обновления архитектурного репозитория

Используй этот навык, когда архитектурный репозиторий уже существует, и его нужно подтянуть к более свежему состоянию продукта без полного анализа с нуля.

## Граница применения

Используй этот skill, когда нужно обновить существующее `as-is` описание:

- появились новые коммиты в основных ветках репозиториев
- изменились контракты, интеграции или хранилища
- появились новые или измененные пользовательские сценарии
- предыдущий анализ устарел и нужно подтянуть только дельту

Не используй его:

- для первого наполнения архитектурного репозитория, в этом случае используй `init-repo-arch-skill`
- для проектирования нового поведения системы, в этом случае используй `new-feature-arch-skill`

## Повторно используемые артефакты

При необходимости используй шаблоны и reference-материалы из [../init-repo-arch-skill](../init-repo-arch-skill/):

- [assets/repo-initialization-progress-template.yaml](../init-repo-arch-skill/assets/repo-initialization-progress-template.yaml)

## Основной workflow

1. Найди существующий архитектурный репозиторий и проверь, есть ли в нем файл прогресса анализа.
2. Если файла прогресса нет, создай его по шаблону [../init-repo-arch-skill/assets/repo-initialization-progress-template.yaml](../init-repo-arch-skill/assets/repo-initialization-progress-template.yaml).
3. Прочитай текущие `landscape`, `hld`, `integrations`, `contracts`, `storage`, `features` и зафиксируй, на каких ветках и коммитах был основан прошлый анализ.
4. Обнови репозитории продукта до актуальных коммитов их основных веток и зафиксируй новый baseline.
5. Определи дельту: какие репозитории изменились, какие контракты и интеграции затронуты, какие новые сценарии или хранилища появились.
6. Проходи только по изменившимся или подозрительным репозиториям, но при этом сохраняй общую связность архитектурной картины.
7. Обновляй существующие артефакты, а не пересобирай всё заново без причины.
8. После каждого существенного обновления синхронизируй файл прогресса: что проверено, что обновлено, что требует дополнительной валидации.
9. Проведи промежуточную валидацию: не расходятся ли новые факты с уже существующим `architecture/`.
10. Если дельта не объясняет наблюдаемое поведение, подключи дополнительные источники: git history, test-repo, runtime-наблюдение, Docker-старт.
11. Если после этого остаются пробелы, задай пользователю только вопросы по изменениям и новым расхождениям, а не повторяй уже закрытые темы.
12. Проведи финальную валидацию и обнови файл прогресса итоговым статусом.

## Правила работы

- Сначала сравни старый baseline и новый baseline, потом обновляй документы.
- Не переписывай весь arch-repo, если изменился только локальный кусок системы.
- Если старый артефакт больше не соответствует коду, не замалчивай это: обнови документ и явно отметь, что именно устарело.
- Если изменения тянут за собой новое поведение, но это уже `to-be`, передавай такую часть в `new-feature-arch-skill`.
- Если изменившийся репозиторий не был включен в прошлый scope, явно зафиксируй расширение scope анализа.

## Ожидаемые результаты

- обновленный файл прогресса анализа
- обновленные baseline-коммиты по репозиториям
- список изменившихся репозиториев и затронутых артефактов
- обновленные `architecture/` и `features/` только в затронутых местах
- список новых пробелов, если изменения нельзя полностью восстановить из кода
