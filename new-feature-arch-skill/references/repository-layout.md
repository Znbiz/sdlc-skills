# Как раскладывать файлы в репозитории

Используй эту структуру, если в репозитории нужно хранить продуктовую и архитектурную документацию по изменениям.

## Базовая структура

```text
repo/
├── README.md
├── features/
│   └── 0001-example-feature.md
├── architecture/
│   ├── constraints.md
│   ├── requirements.md
│   ├── hld.md
│   ├── landscape.yaml
│   ├── tech-stack.md
│   ├── integrations/
│   │   ├── overview.md
│   │   └── <integration>.md
│   ├── contracts/
│   │   ├── <sync-api>.yml
│   │   └── <async-events>.yml
│   └── storage/
│       └── <storage>.yml
```

Шаблоны не нужно хранить в продуктовой документации. Их надо брать из самого skill и использовать только для генерации или заполнения целевых файлов.

## Как раскладывать документы

### `features/`

Сюда клади документы уровня продуктовой фичи или change request.

Один файл на одну значимую фичу или одно значимое изменение поведения.

Обычно файл должен содержать:

- user story или бизнес-сценарий
- функциональные требования
- нефункциональные требования
- ограничения
- основной сценарий
- альтернативные сценарии
- sequence diagram или flow в `mermaid`
- открытые вопросы

Рекомендуемый нейминг:

- `0001-customer-onboarding.md`
- `0002-change-credit-limit.md`

### `architecture/constraints.md`

Сюда клади общие ограничения решения:

- бизнес-ограничения
- технические ограничения
- организационные ограничения
- регуляторные ограничения

Это не файл одной фичи, а агрегирующий документ уровня решения или продукта.

### `architecture/requirements.md`

Сюда выноси агрегированные требования к решению:

- ключевые функциональные требования
- SLA, latency, throughput
- security
- observability
- reliability

Если требования относятся только к одной фиче и не влияют на общую архитектуру, сначала фиксируй их в `features/`, а потом поднимай сюда только то, что важно на уровне системы.

### `architecture/hld.md`

Сюда клади high-level design:

- границы решения
- компоненты
- потоки вызовов и событий
- ключевые архитектурные решения
- риски
- открытые вопросы

Если изменение небольшое, обновляй существующий `hld.md`. Если изменение очень крупное, можно внутри `architecture/` дополнительно завести отдельный HLD-файл, но базовый `hld.md` должен оставаться точкой входа.

### `architecture/integrations/`

Сюда клади описание внешних и межсистемных интеграций.

Используй:

- `overview.md` для обзора всех интеграций
- отдельные `*.md` файлы для сложных интеграций

Отдельный файл нужен, если:

- интеграция критична для бизнеса
- есть сложная обработка ошибок
- есть несколько сценариев обмена
- нужно отдельно описывать ограничения, SLA, retries, idempotency и совместимость

### `architecture/landscape.yaml`

Сюда клади карту ландшафта решения:

- сервисы
- хранилища
- очереди и топики
- внешние системы
- связи между ними

Этот файл должен отвечать на вопрос, какие технические сущности участвуют в целевом решении и как они связаны.

Для каждого сервиса фиксируй краткое поле `technology` в формате `<runtime> / <framework>`.

### `architecture/tech-stack.md`

Сюда клади детальную сводку по ключевому стеку и зависимостям, которые затрагиваются изменением.

Используй этот файл для архитектурно значимых технологий:

- язык и runtime
- основной framework
- ORM и доступ к данным
- transport и integration libraries
- broker clients
- observability
- security и auth библиотеки
- важные внешние SDK

Не превращай этот файл в выгрузку всех пакетов из lockfile. Фиксируй только то, что влияет на понимание решения и сопровождение изменения.

### `architecture/contracts/`

Сюда клади машиночитаемые контракты.

Используй:

- OpenAPI для синхронных API
- AsyncAPI для событий и асинхронных взаимодействий

Рекомендуемый нейминг:

- `customer-profile-api.yml`
- `onboarding-events.yml`

Если контрактов много, держи принцип: один файл на один публичный интерфейс или доменный набор событий.

### `architecture/storage/`

Сюда клади описание хранилищ и структур данных:

- таблицы
- коллекции
- индексы
- ключи
- схемы
- назначение полей

Один файл обычно соответствует одному хранилищу или одной логической схеме.

## Правила раскладки

- Не смешивай продуктовое описание фичи и HLD в одном файле.
- Не клади описание интеграций внутрь `hld.md`, если интеграция уже требует собственной детализации.
- Не храни контракты внутри Markdown, если они должны быть пригодны для валидации или автогенерации.
- Не описывай структуру БД только в тексте, если ее можно зафиксировать в YAML-формате.
- Если изменение локальное, обновляй существующие агрегирующие файлы, а не плодить новые без необходимости.

## Как действовать при новой задаче

1. Сначала проверь, подтверждена ли текущая `as-is` база для затронутой области.
2. Затем создай или обнови документ в `features/`.
3. Затем обнови `architecture/requirements.md`, если изменились системные требования.
4. Затем обнови `architecture/constraints.md`, если появились новые ограничения.
5. Затем обнови `architecture/hld.md`, если меняется решение, поток или состав компонентов.
6. Затем обнови `architecture/integrations/`, если меняются взаимодействия.
7. Затем обнови `architecture/contracts/`, если меняются API или события.
8. Затем обнови `architecture/storage/`, если меняются данные или хранилища.
9. Затем при необходимости актуализируй `architecture/landscape.yaml`.
10. Если изменение затрагивает стек или ключевые зависимости, обнови `architecture/tech-stack.md`.
