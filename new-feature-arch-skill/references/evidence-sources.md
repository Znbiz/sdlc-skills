# Источники фактов для проектирования изменения

Используй этот список, чтобы проектировать будущее состояние на подтвержденной базе, а не на догадках о том, как система работает сейчас.

## Сильные источники

Этим источникам доверяй в первую очередь:

- текущие маршруты API, gRPC handlers, message consumers and producers
- интеграционные тесты и e2e-тесты
- миграции БД, схемы таблиц и текущие модели данных
- машиночитаемые контракты: OpenAPI, AsyncAPI, protobuf
- deployment-конфигурации, Helm values, Terraform, compose, k8s manifests
- код оркестрации бизнес-процесса
- существующие feature flags, migration jobs и rollout-конфиги

Если сервис можно безопасно поднять локально, используй также runtime-наблюдение как сильный дополнительный источник фактов:

- запуск сервиса или набора сервисов в Docker или Docker Compose
- доступные health/readiness endpoints
- реально поднятые HTTP/gRPC endpoints
- фактические топики, очереди, фоновые процессы и зависимости, которые проявляются при старте
- runtime-конфигурация, если она читается из локального окружения и контейнерных настроек

## Средние источники

Эти источники полезны, но требуют перепроверки:

- README и markdown-документы
- `.env.example`, sample configs
- unit-тесты
- диаграммы без признаков актуальности
- описания в задачах и RFC без подтверждения в коде для `as-is` части

## Слабые источники

Используй только как вспомогательный сигнал:

- названия папок без кода
- комментарии без подтверждения в логике
- устаревшие wiki-описания
- устные предположения о том, "как это, наверное, работает"

## Как интерпретировать сигналы

### Текущее состояние

Если вопрос относится к тому, как система работает сейчас, опирайся на код и эксплуатационные артефакты. Не делай проектное решение под неверное представление об `as-is`.

### Будущее состояние

Если вопрос относится к тому, как система должна работать после изменения, различай:

- `fact` для подтвержденного текущего поведения или ограничения
- `inference` для обоснованного вывода из артефактов
- `decision` для выбранного проектного решения
- `assumption` для гипотезы, которую еще нужно проверить

### Runtime-наблюдение

Когда это возможно и безопасно:

- поднимай сервисы локально
- используй запуск как способ подтвердить текущее поведение, состав зависимостей и реальные точки входа
- сверяй runtime-наблюдения с кодом, контрактами и конфигурацией

Не считай успешный старт сервиса автоматическим доказательством корректности проектного решения. Это источник подтверждения `as-is`, а не замена архитектурному выбору.

## Где искать по типам изменений

### Фичи и пользовательские сценарии

Ищи:

- точки входа пользователя или внешнего клиента
- текущие бизнес-операции и статусы
- orchestration-код и use cases
- end-to-end сценарии в тестах
- существующие ограничения и ручные процедуры

### Интеграции

Ищи:

- HTTP clients
- Kafka/Rabbit/NATS producers and consumers
- S3/SFTP connectors
- внешние SDK
- retry/circuit breaker/timeouts
- текущие версии payload и схем

### Контракты

Ищи:

- OpenAPI/Swagger
- protobuf
- AsyncAPI
- DTO/serializers/request-response models
- message payload builders and parsers

### Хранилища

Ищи:

- ORM models
- SQL migrations
- repository implementations
- cache keys and TTL logic
- topic names and retention settings
- текущие правила записи и чтения

## Правило фиксации

Если утверждение не подтверждено явным артефактом, помечай его как:

- `выведено косвенно`
- `требует подтверждения`
- `не подтверждено кодом`

Если утверждение относится к будущему решению, но выбрано сознательно, помечай его как:

- `проектное решение`
- `допущение до подтверждения`
