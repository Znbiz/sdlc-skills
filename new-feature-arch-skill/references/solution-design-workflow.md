# Проектирование решения

Используй этот workflow, когда требование уже достаточно понятно для проектирования целевого решения.

Перед началом убедись, что:

- `as-is` база хотя бы минимально подтверждена
- границы изменения зафиксированы
- ключевые правила и инварианты не висят в воздухе
- есть список затронутых компонентов, контрактов и данных

## Шаг 1. Сформулируй целевую возможность системы

Зафиксируй:

- требуемый бизнес-результат
- целевое поведение для пользователя или внешнего клиента
- ключевые инварианты
- явные non-goals

Это нужно, чтобы HLD не уходил в локальную оптимизацию вместо решения задачи.

## Шаг 2. Определи границы решения

Для целевого решения зафиксируй:

- владеющий сервис или модуль
- downstream-зависимости
- upstream-вызовы
- владение данными
- какие компоненты меняются, а какие остаются без изменений

Если после изменения появляется новый владелец данных или новый orchestration layer, это нужно назвать прямо.

## Шаг 3. Спроектируй поток

Пройди по основному и критичным альтернативным сценариям:

- путь запроса
- путь команды
- путь события
- путь записи в хранилище
- путь чтения
- путь обработки ошибок и повторов

Если процесс долгий, отдельно оцени, не должен ли он стать асинхронным.

## Шаг 4. Сравни варианты, если они существенно различаются

Как минимум проверь, можно ли:

- расширить существующий компонент
- добавить слой оркестрации
- вынести логику в асинхронный процесс
- ввести новое хранилище или избежать этого
- использовать текущий контракт с обратной совместимостью или ввести новую версию

Для каждого разумного варианта оцени:

- стоимость внедрения
- связность
- надежность
- совместимость
- операционные риски

## Шаг 5. Зафиксируй контракты и данные

Отдельно определи:

- изменения API
- схемы событий
- изменения базы данных или хранилища
- версионирование и совместимость
- переходный период, если старый и новый формат должны жить одновременно

Если меняется источник истины, отдельно опиши, кто пишет, кто читает, как проходит миграция и как будет идти reconciliation.

## Шаг 6. Зафиксируй интеграции

Обновляй:

- `architecture/integrations/overview.md`
- отдельные файлы по значимым интеграциям

Для каждой новой или меняемой интеграции определи:

- направление
- триггер
- формат полезной нагрузки
- retries
- timeout behavior
- идемпотентность
- owner интеграции

## Шаг 7. Зафиксируй эксплуатационную модель

Определи:

- observability
- безопасность и управление доступом
- deployment sequence
- rollback и reconciliation
- feature flags
- сигналы успеха и деградации

Если rollout нельзя сделать атомарно, отдельно опиши порядок миграции и окно совместимости.

## Шаг 8. Синхронизируй артефакты

После выбора решения обновляй:

- `features/`
- `architecture/requirements.md`
- `architecture/constraints.md`
- `architecture/hld.md`
- `architecture/landscape.yaml`
- `architecture/tech-stack.md`, если меняется стек или ключевые зависимости
- `architecture/integrations/`
- `architecture/contracts/`
- `architecture/storage/`

Не оставляй архитектурно значимые изменения только в одном HLD, если для них есть более подходящий формализованный артефакт.

## Шаг 9. Проведи валидацию промежуточного решения

Проверь:

- согласованы ли `features/` и `hld.md`
- отражены ли затронутые компоненты в `landscape.yaml`
- отражены ли новые или критичные технологии в `tech-stack.md`
- вынесены ли интеграции из общего текста в `architecture/integrations/`
- есть ли машиночитаемые контракты для значимых API и событий
- описаны ли хранилища и владение данными
- описан ли rollout для неатомарных изменений

Если валидация вскрыла расхождения, сначала исправь артефакты и только потом финализируй решение.

## Шаг 10. Зафиксируй компромиссы и открытые риски

Отдельно перечисляй:

- принятые компромиссы
- отложенные решения
- риски, требующие контроля на delivery
- вопросы, которые не закрываются без дополнительного исследования или согласования

## Чеклист компромиссов

Явно проходи по этим пунктам, когда они релевантны:

- задержка vs надежность
- консистентность vs автономность
- переиспользование vs связность
- простота синхронного подхода vs устойчивость асинхронного
- локальная оптимизация vs сопровождаемость продукта в целом

## Признаки проблем

Остановись и явно обозначь проблему, если видишь что-то из этого:

- больше одного источника истины без стратегии сверки
- несовместимое изменение контракта без плана миграции
- скрытая связность через общие таблицы или неформальные payload
- асинхронный поток без идемпотентности или сценария повторного воспроизведения
- бизнес-критичное решение, делегированное внешней зависимости без fallback
